# Compiler Principles - Semantic Analysis & Symbol Tables Study Notes

## Chapter 5: Semantic Analysis

### Overview
The semantic analysis phase of a compiler:
- Connects variable definitions to their uses
- Checks that each expression has a correct type
- Translates the abstract syntax into a simpler representation suitable for generating machine code

---

## 5.1 Symbol Tables

### Definition
Symbol tables (also called environments) are data structures that map identifiers to their types and locations during semantic analysis.

- **Declarations of identifiers**: IDs are bound to "meanings" in the symbol tables
- **Uses of identifiers**: Look up IDs in the symbol tables
- An environment is a set of bindings denoted by the ↦ arrow
- Example: σ₀ = {g ↦ string, a ↦ int}

### Key Concepts
- Each local variable has a scope in which it is visible
- When semantic analysis reaches the end of each scope, identifier bindings local to that scope are discarded
- Bindings in the right-hand table override those in the left: X + Y ≠ Y + X for tables

---

## Symbol Table Implementation Approaches

### 1. Functional Style Symbol Tables

**Definition**: Keep existing symbol tables unchanged while creating new ones.

**Characteristics**:
- Keep σ₁ in pristine condition while creating σ₂ and σ₃
- Easy to restore σ₁
- Do not modify the original table
- Create new tables by computing the "sum" of existing table and new binding

**Implementation using Binary Search Trees**:
- Can perform "functional" additions efficiently
- When adding a new node at depth d, create d new nodes but don't copy the whole tree
- Copy nodes from root to parent of inserted node
- Original table (m₁) keeps unchanged
- Looking up an element: O(log n) for balanced tree of n nodes

**Why not Hash Tables for Functional Style?**
- Hash tables don't work well for functional style
- Example: m₂ = m₁ + {mouse ↦ 4} where m₁ = {bat ↦ 1, camel ↦ 2, dog ↦ 3}
- If index(camel) = index(mouse) = 5, inserting mouse would overwrite camel
- Copying the entire array is inefficient since hash table arrays should be quite large

### 2. Imperative Style Symbol Tables

**Definition**: Modify existing symbol tables directly with ability to undo changes.

**Characteristics**:
- Modify σ₁ until it becomes σ₂
- While σ₂ exists, cannot look things up in σ₁
- When done with σ₂, undo modifications to get σ₁ back
- Uses a single global environment + an undo stack

**Implementation using Hash Tables**:
- Efficient lookup for large programs with thousands of identifiers
- Uses hash table with external chaining for collision resolution
- Supports easy deletion through stack-like insertion and pop operations

**Insertion Operation Differences**:
- **Functional**: Creates new table without modifying original
- **Imperative**: Modifies existing table but maintains undo capability

When σ + {a ↦ τ₂} and σ contains a ↦ τ₁:
- Insert function leaves a ↦ τ₁ in bucket and puts a ↦ τ₂ earlier in list
- hash(a) -> <a, τ₂> -> <a, τ₁>
- When pop(a) is done at end of a's scope, σ is restored
- hash(a) -> <a, τ₁>

## Code Implementations

### Efficient Imperative Symbol Tables - Basic Hash Table Structure

`*3c
struct bucket { 
    string key; 
    void *binding; 
    struct bucket *next; 
};

#define SIZE 109
struct bucket *table[SIZE];

unsigned int hash(char *s0) {
    unsigned int h=0; 
    char *s;
    for(s=s0; *s; s++)
        h=h*65599 + *s; 
    return h; 
}

struct bucket *Bucket(string key, void *binding, struct bucket *next) {
    struct bucket *b=checked_malloc(sizeof(*b));
    b->key = key; 
    b->binding = binding; 
    b->next = next;
    return b; 
}
`*3

**Explanation**: 
- Defines basic hash table structure with external chaining
- Hash function uses polynomial rolling hash: h = (c₁×65599^(n-1) + c₂×65599^(n-2) + ... + cₙ)
- Bucket constructor creates linked list nodes for collision resolution

### Hash Table Operations

`*3c
void insert(string key, void *binding) {
    int index=hash(key)%SIZE;
    table[index]=Bucket(key, binding, table[index]); 
}

void *lookup(string key) {
    int index=hash(key)%SIZE;
    struct bucket *b;
    for (b = table[index]; b; b=b->next) 
        if (0==strcmp(b->key,key)) 
            return b->binding; 
    return NULL; 
}

void pop(string key) { 
    int index=hash(key)%SIZE;
    table[index]=table[index].next; 
}
`*3

**Explanation**:
- **insert**: Adds new binding at head of collision chain (shadows older bindings)
- **lookup**: Linear search through collision chain, returns first match
- **pop**: Removes head of collision chain (stack-like LIFO behavior)

### Scope Management

`*3c

void S_beginScope(S_table t) { 
    S_enter(t, &marksym, NULL); 
}

void S_endScope(S_table t) {
    S_symbol s;
    do 
        s= TAB_pop(t); 
    while (s != &marksym);
}

**Explanation**:
- **S_beginScope**: Pushes special marker onto auxiliary stack
- **S_endScope**: Pops symbols until finding topmost marker

## Multiple Symbol Tables

In some languages, several active environments exist simultaneously:
- Each module, class, or record has its own symbol table

### Java Example (Forward References Allowed)
`*3java
package M;   
class E {   
    static int a = 5;   
}  
class N {   
    static int b = 10;   
    static int a = E.a + b;  
}   
class D {   
    static int d = E.a + N.a;  
}
`*3

Symbol table construction:
- σ₁ = {a ↦ int}
- σ₂ = {E ↦ σ₁}
- σ₃ = {b ↦ int, a ↦ int}
- σ₄ = {N ↦ σ₃}
- σ₅ = {d ↦ int}
- σ₆ = {D ↦ σ₅}
- σ₇ = σ₂ + σ₄ + σ₆

All classes compiled in environment σ₇. Result: {M ↦ σ₇}

### ML Example (No Forward References)
`*3ml
structure M = struct 
   structure E = struct 
      val a = 5;

      val b = 10

   end

`*3

- N compiled using environment σ₀ + σ₂
- D compiled using environment σ₀ + σ₂ + σ₄
- Result also: {M ↦ σ₇}
